WHAT IS THIS LANGUAGE

DEV
There are TWO equality operators. == is loose. === is strict. One of them you should basically never use.
KABIR
Which one do we actually use?
ARJUN
Always ===. Always. Never ==. Pretend == doesn't exist unless you have a very specific reason.
Strict === vs Loose ==
Click any row to see WHY it evaluates that way.
== LOOSE — coerces first
7 == "7" TRUE
JS converts "7" (string) → 7 (number), then compares 7 === 7. Different types but == doesn't care — it forces them equal.
0 == false TRUE
false → 0 (boolean to number). Then 0 === 0 is true. This is why form validation breaks when checking for empty fields with ==.
null == undefined TRUE
Special case rule in the == algorithm: null and undefined are ONLY equal to each other (and themselves) under ==. This one is actually useful sometimes.
"" == false TRUE
false → 0, then "" → 0, then 0 === 0. Empty string and false both coerce to 0. This kind of chain is why == causes bugs.
[] == false TRUE
An empty array is truthy... but == converts it: [] → "" → 0, and false → 0. So 0 === 0. Mind-bending? Yes. Use === instead.
NaN == NaN FALSE
NaN is the only value NOT equal to itself — in both == and ===. Use Number.isNaN() to check for NaN.
=== STRICT — no coercion
7 === "7" FALSE
Different types (number vs string). No conversion happens. Stops immediately: types don't match → false.
0 === false FALSE
Number vs boolean. Different types. === never converts. This is the safe, predictable result you actually want.
null === undefined FALSE
Both are "empty" but they ARE different types. null is object-typed, undefined is undefined-typed. Strict: different types → false.
7 === 7 TRUE
Same type (number), same value (7). === simply checks: type match? ✓. Value match? ✓. Result: true. Clean, predictable.
"hi" === "hi" TRUE
Same type (string), same content. Strings are compared character by character. Identical → true.
{} === {} FALSE
Two separate objects in memory. === for objects compares REFERENCES (memory addresses), not content. They look the same but they're different objects. Same rule for arrays.

— TOPIC 4: LOGICAL OPERATORS & SHORT-CIRCUIT —
DEV
The most USEFUL hidden feature of && and || is short-circuit evaluation. Most people miss this completely.
KABIR
Short-circuit? What does that mean?
DEV
If && finds a falsy value on the LEFT, it stops. Never looks right. If || finds a truthy value on the LEFT, it stops. That's how you safely check for null without crashing.
Logical Operators — Real Usage
⚡ && AND
True only if BOTH sides truthy. Stops at first falsy — never evaluates right side.
let user = null;
// Safe — stops at user (null)
user && user.name === "Admin"
// no crash ✓
✅ || OR
True if EITHER is truthy. Stops at first truthy — perfect for default values.
let name = ""; // falsy
let display = name || "Guest";
// display = "Guest" ✓
// great default value pattern
🔄 ! NOT
Inverts truthy/falsy. Coerces to boolean first, then flips. !! (double not) converts anything to boolean.
!true // false
!0 // true
!"" // true
!!"hi" // true (toBool)

— TOPIC 6: RELATIONAL COMPARISONS < > — THE WEIRD RULES —
SCENE 05 The Rules That Break Your Brain 📍 Priya Finds Something in the Console

Both sides are STRINGS. So JS does dictionary comparison. "9" comes after "10" alphabetically — same as "banana" > "apple".
Relational Comparison Rules
Click any case to see exactly which rule triggered. Two sets of rules: string vs string = dictionary. Everything else = numeric conversion.
📖 String vs String → Dictionary
"10" > "9" → false string
Both strings → dictionary order. Compare first character: "1" vs "9". Character codes: "1" = 49, "9" = 57. 49 < 57, so "10" < "9".
"b" > "a" → true string
Both strings → dictionary. "b" (98) > "a" (97) in char codes. This is lexicographic comparison — same as how a dictionary orders words.
"apple" < "banana" → true string
Compare "a" vs "b" at position 0. "a"(97) < "b"(98). Since the first characters differ, result decided immediately.
"Z" < "a" → true string
Uppercase letters have LOWER char codes than lowercase. "Z" = 90, "a" = 97. So all uppercase letters are "less than" all lowercase letters.
🔢 Mixed Types → Both Become Numbers
"5" > 3 → true numeric
One is a number → numeric conversion. "5" → 5 (number). Then 5 > 3 → true. When types are mixed in <>, both get converted to numbers.
null > 0 → false numeric
null → 0 (numeric conversion). Then 0 > 0 → false. Surprising: null == 0 is also false (special case). null only equals undefined with ==.
undefined > 0 → false numeric
undefined → NaN (numeric conversion). Any comparison with NaN is ALWAYS false. NaN > 0, NaN < 0, NaN == 0 — all false.
true > false → true numeric
Both booleans → numbers. true = 1, false = 0. Then 1 > 0 → true. Booleans in comparisons always convert to numbers first.
🌳 The Comparison Decision Tree
Dev draws it on paper. Arjun copies it to notes. Kabir actually learns it.
❓ Which operator?
├──
→ === or !==
↓
Check types first. Different types? Immediately false. Same types? Compare values. For objects: compare references (memory addresses). NaN === NaN is ALWAYS false.
├──
→ == or !=
↓
Same types? → use === rules. null == undefined? → true (special). Boolean involved? → convert bool to number, restart. String + number? → convert string to number. Object + primitive? → convert object (.valueOf then .toString). Otherwise → false.
└──
→ < > <= >=
↓
BOTH sides are strings? → Dictionary (character code) comparison. ANY other combination? → Convert BOTH to numbers (null→0, undefined→NaN, booleans→0/1). If either is NaN → always false.

📓 Chapter 3 — Summary
📖 Group Notes — Four Contributors
➕
Arithmetic: +, -, *, /, %, ** — modulo (%) checks even/odd, finds cycles. i++ (postfix) returns THEN increments. ++i (prefix) increments THEN returns.
📝
Assignment shorthands: +=, -=, *=, /=, %=, **= — all mean "do the operation and store back." Cleaner than writing x = x + 5.
⚠️
ALWAYS === , NEVER ==. Loose == performs type coercion: 7 == "7" is true, 0 == false is true. Strict === never converts: different types → immediately false.
🔄
Explicit conversion: Number(), String(), Boolean() — use the type name as a function. parseInt("100px") → 100. Number("100px") → NaN. Know the difference.
✅
6 falsy values ONLY: false, 0, "", null, undefined, NaN. Everything else is truthy — including [], {}, and "false". Check arr.length not arr.
⚡
Short-circuit: && stops at first falsy (safe null checks). || stops at first truthy (default values). !! converts anything to boolean.
❓
Ternary: condition ? trueValue : falseValue — one-line if-else. Don't nest them. Relational <>: both strings → dictionary. Mixed types → both become numbers.
💡
Float mismatch: 0.1 + 0.2 ≠ 0.3 because binary can't represent most decimals exactly. The tiny storage errors accumulate. Fix: integers for money, toFixed for display.
